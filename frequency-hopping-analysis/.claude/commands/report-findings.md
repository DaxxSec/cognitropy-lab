# /report-findings — Generate Bayesian FHSS Findings Report

Produce a structured report that captures every claim about the analysed system as a posterior with explicit prior, likelihood, and credibility interval.

## Inputs

- Latest engagement directory: `outputs/<YYYY-MM-DD>-<engagement-name>/`
- Calibration evidence (the calibration directory referenced in `calibration.json`)
- Optional: `--audience <engineer|exec|legal>` to set tone

## Steps

### 1. Pull all per-step outputs
- `hop-detect.json`
- `dwell-estimate.json`
- `sequence-id.json`
- `jammer-flag.json`
- `dehop/summary.json`
- Calibration evidence

If any are missing, list them as "not run" — do not fabricate.

### 2. Validate calibration
- Confirm a calibration ran in the same session.
- Confirm calibration succeeded (Bluetooth Classic posterior tests passed).
- If calibration is missing or stale (> 24 h old), the report's CI claims are downgraded to "approximate" and a banner is added to the top.

### 3. Structure the report

```markdown
# FHSS Findings — <System Name>
**Engagement:** <name> | **Date:** <YYYY-MM-DD> | **Analyst:** <user>
**Calibration:** <PASS / DOWNGRADED / FAIL — link to calibration evidence>
**Authorisation:** <link to authorisation document on file>

## Summary
One paragraph. Lead with the headline finding; do not lead with method.

## System Characterisation

| Parameter | Posterior Mean | 95% CI | Prior | Method |
|-----------|----------------|--------|-------|--------|
| Hop rate (hops/s) | ... | [..., ...] | log-uniform / system-specific | Forward-backward + NUTS |
| Channel count K | ... | [...,...] | uniform on legal grid + filings | NUTS |
| Dwell time (µs) | ... | [..., ...] | Gamma weak | Edge detection + NUTS |
| Sequence type | ... (P=...) | — | Categorical([0.4,0.3,0.2,0.1]) | Likelihood comparison |

## Sequence Recovery

- Recovered parameters: …
- Predictability horizon: …
- Security implications: …

## Jammer / Interference Activity

- Narrowband jammers detected: …
- Sweep jammers detected: …
- Follower jammer suspected: P=…
- Per-dwell jamming fraction: …

## Limitations

- Capture coverage: <bandwidth, duration>. Hard limits per `domain-knowledge.md` §7.
- SNR floor: …
- Posterior modes / multimodality: …
- Anything that could be wrong: …

## Recommendations

- For the system operator: …
- For follow-up captures: …
- For responsible disclosure: …

## Reproducibility

- Capture: `<path>` + SigMF metadata
- Posterior: `<path>` (arviz NetCDF)
- Code version: `<git sha of this workspace>`
- PyMC version: `<pymc.__version__>`
```

### 4. Audience-specific tweaks
- `engineer`: Numbers + CIs, no executive summary, full reproducibility section.
- `exec`: One-paragraph summary, key risks, no math.
- `legal`: Chain of custody emphasised, authorisation document linked, every claim attributed to a posterior.

### 5. Output

Write to `outputs/<engagement>/report.md`. Also append a one-line summary to `user-docs/report.md` so the user has a cumulative log.

If the report includes vulnerability-grade findings (predictable PN seed, weak AFH, follower-jammer detection), automatically open a section listing the next responsible-disclosure steps with vendor + CERT contacts.

### 6. Sign

Append:
```
---
Generated by `frequency-hopping-analysis` workspace v1.0
Calibration: <PASS/DOWNGRADED/FAIL>
Bayesian model documented in context/for-agent/domain-knowledge.md
```

## Output

A reproducible, audience-tuned, calibration-validated Bayesian report. The report is the deliverable.
