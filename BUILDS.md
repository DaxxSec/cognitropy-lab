## How to read this log

- **Day** — sequential count of days since project start (`2026-03-26`). NOT the count of builds, because the v2 engine produces two builds per day (morning + evening) starting 2026-05-20 — so Day 56 has two entries from that point onward.
- **Slot** — `morning` or `evening`. Pre-doubling entries default to `morning` in the schema; their effective slot was the single daily build at 06:05.
- **Type** — `Standard` for single-domain workspaces; `**× SecondaryCategory**` for crossover days where the engine rolled a secondary domain and emitted a "spark" prompt fusing the two. About 30% of slots roll crossover.
- **Domain** — the effective primary domain that got built (after the anti-clustering step in the daily-build pipeline). On crossover days, the secondary domain is shown inline in italics.
- **Technique** — the engine's "technique modifier" for that slot; same 30-modifier pool drives every build but each slot rolls it independently. The technique often produces the most surprising framing — e.g. RF spectrum analysis via *chronic-patient triage* (May 12), or hybrid powertrains via *Bayesian probability assessment* (May 9 + May 16).

## Note on chronology vs build count

Day numbering reflects calendar days since project start, not build count, because some early days have no entry (engine fired but build aborted, or scheduled task didn't run) and starting 2026-05-20 days carry two slots. To count builds in a window, count rows; to count days, count distinct dates.
